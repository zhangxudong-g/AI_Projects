## 1. 概要概述

| 項目 | 内容 |
|------|------|
| **ビュー名** | `GKBVEUCDVTAISHO` |
| **主目的** | 学齢簿（児童・保護者）から、現在有効な DV（データベース・バリデーション）対象者を抽出し、氏名・生年月日・在籍学校情報などを一括で取得する |
| **利用シーン** | - 学校・教育委員会向けのレポート生成<br>- データ連携（例：WizLIFE 2次開発）<br>- 法令対応（例：CTC.GL UPDATE 1.0.010.001） |
| **主要テーブル** | `GKBTGAKUREIBO`（学齢簿）<br>`GABTATENAKIHON`（宛名基本：児童・保護者）<br>`GACTIDOHAKKOYOKUSHIKANRI`（DV管理マスタ）<br>`GKBT*` 系列（学校マスタ） |
| **変更履歴のポイント** | - 2024/06/03: WizLIFE 2次開発に伴う DV 期間取得ロジック変更（`least`/`greatest`）<br>- 2025/07/15: CTC.GL の QA 対応で削除フラグ `SAKUJO_FLG = 0` を追加 |

> **新規開発者へのメッセージ**  
> このビューは「現在有効な」児童・保護者レコードだけを抽出するため、日付条件や外部結合の有無が結果に大きく影響します。変更時は **日付ロジック** と **外部結合 (+)** の挙動を必ずテストしてください。

## 2. コード級洞察

### 2.1 カラム構成（重要ポイント）

| カラム | 説明 | コメント |
|--------|------|----------|
| `児童生徒宛名番号` | 学齢簿の児童キー (`KOJIN_NO`) | 主キー相当 |
| `児童生徒氏名かな/漢字` | 児童氏名（かな・漢字） | `ATENAJ` から取得 |
| `児童生徒生年月日` / `表示児童生徒生年月日` | 生年月日（内部用・表示用） | `ATENAJ.SEINENGAPI` 系列 |
| `児童生徒DV期間から` / `児童生徒DV期間まで` | DV 有効期間（最小開始日・最大終了日） | `least` / `greatest` で 3 つの期間を統合 |
| `保護者宛名番号`・`保護者氏名かな/漢字` | 保護者情報 | `ATENAH` 系列 |
| `保護者DV期間から` / `保護者DV期間まで` | 保護者の DV 期間 | 同上 |
| `学年`・`小学校コード/名`・`中学校コード/名`・`区域外学校コード/名`・`特別支援学校コード/名` | 学校情報 | 各マスタテーブルと外部結合 |
| `国私立学校コード/名` | 2024 追加項目（国・私立学校） | `KUNISIRITSUGAKKO` から取得 |

### 2.2 主なロジックフロー

```mermaid
flowchart TD
    A["開始: SELECT 文"] --> B{"JOIN 条件"}
    B -->|"学齢簿 ↔ DV管理 (児童)"|
        C["外部結合: GAKUREIBO.KOJIN_NO = DVKANRI_JIDO.KOJIN_NO(+)"]
    B -->|"学齢簿 ↔ DV管理 (保護者)"|
        D["外部結合: GAKUREIBO.HOGOSYA_KOJIN_NO = DVKANRI_HOGOSYA.KOJIN_NO(+)"]
    B -->|"学齢簿 ↔ 宛名基本 (児童)"|
        E["内部結合: GAKUREIBO.KOJIN_NO = ATENAJ.KOJIN_NO"]
    B -->|"学齢簿 ↔ 宛名基本 (保護者)"|
        F["内部結合: GAKUREIBO.HOGOSYA_KOJIN_NO = ATENAH.KOJIN_NO"]
    B -->|"学齢簿 ↔ 学校マスタ"|
        G["外部結合: 各コード (+)"]
    C & D --> H["期間取得: least / greatest"]
    H --> I["日付フィルタ: 現在日付が期間内か"]
    I --> J["その他フィルタ: 学年 0-9, 削除フラグ 0, ..."]
    J --> K["ORDER BY"]
    K --> L["結果セット生成"]
    L --> M["終了"]
```

#### 重要ロジック解説

1. **DV 期間統合**  
   - `least(開始1, 開始2, 開始3)` → 3 つの開始日から最も早い日を取得。  
   - `greatest(終了1, 終了2, 終了3)` → 3 つの終了日から最も遅い日を取得。  
   - これにより、児童・保護者それぞれが **いずれかの DV 期間** に該当すれば抽出対象になる。

2. **有効日付判定**  
   ```sql
   least(...) <= TO_NUMBER(TO_CHAR(SYSDATE,'YYYYMMDD'))
   AND NVL(greatest(...),99999999) >= TO_NUMBER(TO_CHAR(SYSDATE,'YYYYMMDD'))
   ```
   - 現在日付が統合期間内にあるかを判定。`NVL(...,99999999)` で終了日が NULL の場合は「無期限」とみなす。

3. **外部結合 (+)**  
   - DV 管理テーブルは **左外部結合** で結合。DV が未設定でも児童・保護者情報は取得できるが、期間カラムは `NULL` になる。  
   - 学校コードは全て外部結合 (`(+)`) で、マスタが無くてもレコードは残す設計。

4. **削除フラグ**  
   - `GAKUREIBO.SAKUJO_FLG = 0` が 2025 の更新で追加。削除済みレコードは除外。

### 2.3 パフォーマンス考慮

| 項目 | 推奨対策 |
|------|----------|
| **インデックス** | `GKBTGAKUREIBO(KOJIN_NO)`, `GKBTGAKUREIBO(HOGOSYA_KOJIN_NO)`, `GACTIDOHAKKOYOKUSHIKANRI(KOJIN_NO)` にインデックスを張ると外部結合が高速化 |
| **日付比較** | `TO_NUMBER(TO_CHAR(SYSDATE,'YYYYMMDD'))` は毎回計算されるので、実行計画上は定数化されるが、可能なら `SYSDATE` のみで比較できる形にリファクタリング検討 |
| **外部結合の数** | 6 つ以上の外部結合があるため、実行計画で `HASH JOIN` が多用される可能性。テーブルサイズが大きい場合はパーティショニングやマテリアライズドビューの活用を検討 |

## 4. 変更・保守時のチェックリスト

1. **日付ロジック変更**  
   - `least` / `greatest` の対象カラムが増減したら、必ず **全パターンのテストケース**（開始日が NULL、終了日が NULL、重複期間など）を追加。  

2. **外部結合追加/削除**  
   - 新しいマスタテーブルを結合する場合は、`(+)` の有無を検討し、**削除フラグ**や**NULL 許容**の影響を評価。  

3. **削除フラグ (`SAKUJO_FLG`)**  
   - 追加後は **データクレンジングバッチ** が正しくフラグを設定しているか確認。  

4. **パフォーマンス回帰テスト**  
   - 大量データ（数百万件）で **実行計画** を取得し、インデックスの有無や `HASH JOIN` の使用状況を確認。  

5. **ドキュメント更新**  
   - カラム追加やコメント変更があったら、**COMMENT ON COLUMN** 部分も忘れずに更新。

## 5. まとめ

`GKBVEUCDVTAISHO` は、児童・保護者の **現在有効な DV 対象者** を抽出し、学校情報まで包括的に取得できる重要なビューです。  
- **日付ロジック** と **外部結合** が結果の正確性を左右するため、変更時は必ずテストとパフォーマンス評価を行ってください。  
- 将来的に **新しい学校マスタ** や **追加の DV 期間項目** が増える可能性があるため、拡張しやすい構造（`least`/`greatest` の利用）を保ちつつ、インデックス設計を見直すことが推奨されます。
