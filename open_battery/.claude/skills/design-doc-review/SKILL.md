# Design Document Review Skill

## 触发条件

当用户提到以下任何内容时，使用此 skill：
- "评审文档"、"review 文档"、"审查设计"
- "检查需求"、"验证方案"、"看看这个文档有什么问题"
- "review this doc"、"check this design"
- 提供 `.md` / `.txt` / `.yaml` 文件并要求评审
- 提到 "设计方案"、"技术方案"、"需求文档"、"PRD" 并需要反馈

## 技能描述

系统性地评审嵌入式固件项目的设计文档、需求规范、技术方案，按多维度检查清单进行分析，输出结构化评审报告。

**适用于**：
- AI 辅助开发工作流文档
- 芯片驱动设计文档
- 需求规范（YAML 格式）
- 状态机设计文档
- 接口规范文档
- 测试策略文档

## 评审维度

### 1. 可行性（Feasibility）
- AI 能力边界是否明确
- 技术方案是否可实现
- 资源约束是否满足（内存、GPIO、中断）
- 时序要求是否可达
- 硬件限制是否考虑

### 2. 完整性（Completeness）
- 所有必需的章节是否存在
- 是否覆盖所有用例/场景
- 错误处理是否完整
- 边界条件是否定义
- 版本管理是否涵盖
- 测试策略是否完整

### 3. 合理性（Rationality）
- 设计原则是否合理
- 状态机逻辑是否正确（无死锁、可达性）
- 时序参数是否明确
- 资源分配是否冲突
- 优先级是否合理
- 错误恢复策略是否有效

### 4. 一致性（Consistency）
- 术语使用是否统一
- 参数定义是否前后一致
- 接口规范与实现是否对应
- 文档间引用是否正确

### 5. 实用性（Practicality）
- 是否提供使用示例
- 是否有检查清单
- 是否有工具支持
- 是否容易理解和执行

### 6. 安全性（Safety）
- 故障模式是否考虑（FMEA）
- 保护机制是否完善
- 恢复策略是否定义
- 中断安全性是否保证

## 评审流程

### 第一步：理解文档类型

识别文档类型并应用对应的检查清单：

**工作流文档**：
- 阶段划分是否清晰
- 输入输出是否明确
- 验证点是否设置
- 工具支持是否充分

**需求规范**：
- 触发条件是否明确
- 前置/后置条件是否完整
- 参数范围是否定义
- 优先级是否标注

**接口规范**：
- 函数签名是否完整
- 参数约束是否明确
- 返回值/错误码是否定义
- 使用示例是否提供

**状态机设计**：
- 初始状态是否定义
- 所有状态是否可达
- 是否存在死锁
- 转换条件是否完整
- 事件优先级是否合理

### 第二步：多维度分析

按照上述 6 个维度逐一检查，记录：
- ✅ 优点与亮点
- ⚠️ 问题与风险（按严重程度分级：P0/P1/P2）
- 💡 改进建议（具体可执行）

### 第三步：生成评审报告

输出结构化报告，包含：

```markdown
## 评审摘要

| 评审维度 | 评分 | 关键发现 |
|---------|------|---------|
| 可行性 | ⭐⭐⭐⭐☆ | ... |
| 完整性 | ⭐⭐⭐☆☆ | ... |
| ...

## 优点与亮点
1. ...
2. ...

## 关键问题（按优先级）

### P0 - 必须修复
1. **问题**：...
   **影响**：...
   **建议**：...

### P1 - 强烈建议
...

### P2 - 可选改进
...

## 具体改进建议

### 建议 1：[标题]
**当前状态**：...
**问题**：...
**改进方案**：...
**示例**：...

## 检查清单

- [ ] 项目 1
- [ ] 项目 2
...
```

## 嵌入式项目特定检查项

### 硬件相关
- [ ] GPIO 资源分配无冲突
- [ ] 中断优先级配置合理
- [ ] 时序参数满足数据手册要求
- [ ] IIC/SPI 地址格式正确（7 位 vs 10 位）
- [ ] 电气特性（上拉/下拉、推挽/开漏）明确

### 软件相关
- [ ] 状态机无死锁，所有状态可达
- [ ] ISR 中无阻塞调用
- [ ] 内存使用在可用范围内
- [ ] 延时函数平台适配正确
- [ ] 边界条件处理明确（> vs >=）

### 错误处理
- [ ] 所有通信操作有超时保护
- [ ] 重试策略定义明确
- [ ] 错误码分类完整
- [ ] 恢复策略可执行
- [ ] 故障模式有应对方案

### 测试相关
- [ ] 测试覆盖率要求明确
- [ ] 边界条件测试完整（阈值 ±1）
- [ ] 故障注入场景定义
- [ ] 长时间稳定性测试要求

## 输出格式

采用 Markdown 格式，使用：
- 表格：对比数据
- 列表：问题清单
- 代码块：示例代码
- emoji：视觉标记（✅⚠️❌💡🔴🟡🟢）
- 引用：强调关键信息

## 示例触发

```
用户: review 这篇文档，从设计的可行性、完整性、合理性等各个方面
Claude: [启动 design-doc-review skill]
```

## 注意事项

1. **客观中立**：既指出问题，也肯定优点
2. **具体可执行**：建议要有具体的改进方案，不能只说"需要改进"
3. **优先级明确**：区分 P0（必须修复）/ P1（强烈建议）/ P2（可选）
4. **示例充分**：问题和建议都要有具体示例
5. **嵌入式特性**：关注硬件约束、实时性、资源限制
