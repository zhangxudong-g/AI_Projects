
## 自定义技能

### 1. Design Document Review（设计文档评审）

**位置**：`.claude/skills/design-doc-review/`

**用途**：系统性地评审嵌入式固件项目的设计文档、需求规范、技术方案。

**触发方式**：
- "review 这篇文档"
- "评审 XXX.md"
- "检查这个设计有什么问题"
- "验证这个状态机设计"

**评审维度**：
1. **可行性**：技术上能否实现，AI 能力边界是否明确
2. **完整性**：是否覆盖所有必要内容
3. **合理性**：设计是否合理，状态机是否有死锁
4. **一致性**：术语和引用是否统一
5. **实用性**：是否易于理解和执行
6. **安全性**：故障和保护机制是否完善

**嵌入式特定检查**：
- GPIO 资源分配冲突检查
- 中断优先级合理性验证
- 状态机死锁检测
- ISR 阻塞调用检查
- 时序参数验证
- 边界条件完整性检查

**使用示例**：
```
# 评审工作流文档
review docs/guides/ai-dev-workflow.md，从可行性、完整性、合理性等方面

# 评审状态机设计
验证这个状态机设计，检查是否有死锁

# 评审需求规范
审查这个 YAML 需求规范，看看有没有遗漏
```

**输出格式**：
- 评审摘要（含评分表）
- 优点与亮点
- 关键问题（按 P0/P1/P2 优先级分级）
- 具体改进建议（含示例）
- 检查清单

## 项目特定配置

### CLAUDE.md

项目根目录的 `CLAUDE.md` 包含：
- 项目概述（芯片架构、开发工具）
- 固件结构说明
- Pin 定义和 LED 指示
- 寄存器定义（SW6301/SW6303）
- C51 编程规范
- IIC 软件模拟时序
- 自我改进记录和待办事项

**重要**：`CLAUDE.md` 会在每次会话开始时自动加载，提供项目上下文。

### 编码规范

本项目遵循以下规范：
- **语言**：Keil C51（8051 内核）
- **命名**：小写 + 下划线（如 `sw6301_read_reg()`）
- **位操作**：使用 `sbit` 定义位变量
- **中断**：使用 `interrupt` 关键字，避免 ISR 中阻塞调用
- **延时**：使用 `_delay_ms()` / `_delay_us()`

## 工作流集成

本项目遵循 **AI 辅助固件开发工作流**（见 `docs/guides/ai-dev-workflow.md`）：

```
阶段1: 芯片资料 → AI → 硬件抽象层 → [人工验证]
阶段2: 用户需求 → AI细化 → 业务逻辑 → [人工审查]
阶段3: 自动化测试 → L1/L2/L3 → [实机验证]
```

**人工验证点**：
- 阶段 1：寄存器定义、时序约束、错误处理
- 阶段 2：状态机逻辑、边界条件、中断安全
- 阶段 3：实机验证、长时间稳定性测试

## 使用技巧

### 1. 利用 Design Review Skill

在修改重要设计文档后，使用 skill 进行自动评审：
```
我刚更新了 ai-dev-workflow.md，review 一下有什么问题
```

### 2. 引用项目上下文

Claude 已加载项目上下文（通过 `CLAUDE.md`），可以直接提问：
```
SW6301 的 IIC 地址是 10 位还是 7 位？
如何处理 IIC 总线卡死的情况？
```

### 3. 请求代码生成

遵循项目规范请求代码：
```
按照项目中的 IIC.c 风格，生成 SW6301 的寄存器读写函数
```

### 4. 状态机验证

请求验证状态机设计：
```
检查这个状态机是否有死锁，所有状态是否可达
```

## 维护

编辑 `CLAUDE.md` 以：
- 添加新芯片资料
- 更新自我改进记录
- 补充最佳实践

- 每次任务完成后复盘
- 分析行为模式
- 识别改进点
- 自动更新 `CLAUDE.md`

查看改进历史：
```
查看 CLAUDE.md 中的"自我改进建议"部分
```
